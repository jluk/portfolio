Karpathy's Autoresearch: Self-Directed AI Research While You Sleep

Justin Luk · March 12, 2026 · 8 min read ·

Andrej Karpathy released a project called autoresearch that lets an AI agent autonomously modify ML training code, run experiments, evaluate results, and iterate — in a loop, unattended, overnight. It actually works and improved a very strong baseline approach.

I think it's one of the more important things to come out of the AI tooling space this year, and it deserves a clear explanation for people who aren't neck-deep in ML Twitter. It will certainly be cited as obvious in hindsight, a mark of a great product.

What is it

The setup is relatively straightforward. There's a training script that builds and trains a small language model. An AI agent is given permission to modify that script — and only that script. Then it loops:

  1. Modify the code — architecture, hyperparameters, optimizer, model depth. Anything in the training script is fair game.
  2. Train for five minutes — defined by wall-clock time, not steps or epochs. This matters and I'll explain why.
  3. Measure validation loss — specifically bits per byte (bpb), a metric that's independent of vocabulary size, so you can compare across different tokenizer configurations fairly.
  4. Keep or revert — if bpb improved, the change is committed. If not, git reset. Clean slate.
  5. Repeat — the agent's instructions literally say "NEVER STOP."

You get twelve experiments per hour. Roughly a hundred overnight. You go to sleep and wake up to a log of everything the agent tried and a model that's measurably better than where it started.

Not your grandpa's hyperparameter tuning

Automated hyperparameter search has existed for decades. Grid search, random search, Bayesian optimization — all well-understood. But those tools search within a space that a human defines. You set bounds on learning rate, batch size, maybe model width, and the system explores that finite box.

Autoresearch is very different because the agent modifies source code. It can restructure the model architecture, introduce new techniques, change the optimizer, or rewrite the training loop. It's not bounded by a predefined search space. It operates the way a researcher operates — read the code, form a hypothesis, change something, measure the outcome.

The gap between "search within bounds I set" and "modify the code itself" is the gap between optimization and research.

You have five minutes, Mr. Agent

If you measure experiments by training steps, your results are hardware-dependent. An H100 and a laptop would yield incomparable data. The five-minute wall-clock constraint makes results portable across hardware so the agent discovers what's optimal for whatever machine it's on.

It also creates a useful pressure on efficiency. A bloated model that trains slowly will lose to a lean model that converges fast. This is a real constraint in production ML — compute has a cost, and speed of iteration compounds. This time budget encodes that reality directly into the experiment loop.

And practically: five minutes means failed experiments are quite cheap. The agent can take risks and it should as most ideas in ML research don't work. A system that can try a hundred of them overnight, cheaply, and keep only the wins produces high expected value with low downside per trial.

Training code is Karpathy level

Worth noting: the code the agent modifies is legit as you'd expect from Karpathy. The whole thing is ~630 lines of Python — minimal, but state-of-the-art.

Here's what the baseline already does:

  • A smarter way to learn from mistakes. When a neural network trains, it adjusts millions of numbers to get better at its task. The optimizer is the algorithm that decides how to adjust them. Most systems use one strategy for everything. Autoresearch uses a hybrid — one approach for the big matrices of numbers (where the math benefits from a technique called polar orthogonalization), and a different approach for everything else. It's like having a different coaching style for sprinters vs. marathon runners on the same team.
  • Selective memory during reading. When a language model reads text, "attention" is how it decides which earlier words matter for predicting the next one. Full attention — looking at everything that came before — is expensive. Autoresearch alternates layers between full-context and short-context attention, so the model gets both the big picture and local detail without the full computational cost.
  • Zero wasted data. Training data gets chopped into batches. Normally, shorter documents get padded with empty space to fill a batch evenly — like shipping half-empty boxes. Autoresearch packs documents together optimally, hitting 100% utilization. Every token the GPU processes is real data. More learning per compute dollar.
  • Fast-fail on bad experiments. If the model's numbers blow up mid-training (NaN values, exploding loss), training stops immediately instead of burning the remaining five minutes on a doomed run.

The agent is experimenting on a very well-engineered baseline. Improvements it finds are real improvements to SOTA knowledge, not artifacts of a weak starting point with improvements that are well-understood already.

Expected value math

Most experiments will fail and that's great as this system is designed for it.

Failed experiments cost five minutes of GPU time and nothing else — the code reverts automatically. Successful experiments compound as each improvement becomes the new baseline for the following run.

The early results speak for themselves.

  1. Over two days, the agent ran ~700 experiments and found ~20 real improvements. Stacked together, time-to-GPT-2 dropped from 2.02 to 1.80 hours — an 11% speedup. The agent caught oversights in attention scaling and regularization that Karpathy himself had missed. One community member documented their overnight run where 7 of 35 experiments succeeded, and the model improved by getting simpler.
  2. Shopify CEO Tobi Lutke ran it overnight and woke up to an agent-optimized 0.8B model that outscored his previous 1.6B — a 19% improvement in validation scores. His reaction was "OK this thing is totally insane."

If 10% of experiments produce a measurable gain, that's ten improvements overnight which compound on the last. Over a week, that's 70 improvements. While gains may be incremental per experiment, the compounding is significant. This is the same math that makes continuous integration incredibly valuable — small, frequent, validated changes accumulate into large differences. (Thanks calc3, for making me take the area of a donut.)

The cost of failure is bounded and the value of success is cumulative.

Next will be scaling compute, time, memory and application domains

This is a single agent on a single GPU. The logical next steps:

1. Parallel agents across a cluster. Karpathy has already said the next step is SETI@home-style massive collaboration: "The goal is not to emulate a single PhD student, it's to emulate a research community of them." Multiple agents exploring different parts of the design space simultaneously, with a shared leaderboard of what's working. This is where throughput goes from a hundred experiments a night to thousands.

2. Smarter memory-led exploration. Right now the agent proposes changes based on its general knowledge of ML. Future versions will incorporate the history of what's already been tried — what worked, what didn't, and why. This moves the system from "informed trial and error" toward something closer to directed research with memory.

3. Longer budgets for deeper experiments. Some techniques only show their value after extended training. Five minutes is a good starting point for rapid iteration, but expanding to thirty-minute or hour-long runs would let the agent explore changes that need more time to converge.

4. Application beyond language models. The loop structure — modify code, run experiment, measure outcome, keep or revert — is not specific to NLP. It applies anywhere you have a codified experiment and a measurable objective. Protein structure prediction, reinforcement learning reward shaping, compiler optimization. The pattern transfers. Community forks are already proving this out — there's an Apple Silicon MLX port and a Windows RTX fork expanding hardware support.

Conclusions & take-off velocity

Recursive self-improvement (spooky) has been lauded by optimists (golden goose) and doomers (boogeyman). An AI that can make itself better, which makes it better at making itself better, and so on. It's usually framed as a theoretical concern or a far-future scenario.

Autoresearch is honestly a working version of that loop. An AI agent improves the code that trains AI models with compounding results. While the agent doesn't modify its own weights directly — it modifies the training process that produces them. It's not skynet but the feedback loop is here for AI to improve how AI gets built.

The bottleneck in ML research has never been ideas, but time and attention to test them. A researcher can maybe run a handful of meaningful experiments in a day, while this system runs a hundred overnight. This agent already caught optimizations that a brilliant two-decade veteran of ML missed. Not because the agent is smarter — because it has more attempts at-bat.

No, this doesn't replace researchers. It does supercharge their effective output though. Instead of running experiments, you define the bounds for your caffeine-charged intern. You design the loop, set the constraints, and evaluate the trajectory.

This is a for-loop with really good taste. And it's the special kind of project that looks small in the moment and obvious in hindsight. A loop, a metric, and an agent with permission to edit code. The rest is compute and time.
